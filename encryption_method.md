# Encryption Methodology

This document describes the encryption approach used in the Fitness Tracker app to protect user data. The implementation lives in `EncryptionService` (and related auth/database code) and uses Apple’s **CryptoKit** and **CommonCrypto**.

---

## 1. Overview

- **Data encryption:** User fitness/measurement data is encrypted at rest with a **per-user Data Encryption Key (DEK)**.
- **Key protection:** The DEK is itself encrypted with a **Key Encryption Key (KEK)** derived from the user’s password, and the encrypted DEK is stored in the database.
- **Password verification:** Passwords are hashed (with a salt) for verification only; the same password is used to derive the KEK that protects the DEK.

---

## 2. Key Hierarchy

| Key | Role | Size / derivation | Stored / in memory |
|-----|------|-------------------|--------------------|
| **KEK** (Key Encryption Key) | Protects the DEK | 256-bit, from password via PBKDF2 | Not stored; derived at login from password + key salt |
| **DEK** (Data Encryption Key) | Encrypts/decrypts measurement data | 256-bit random (AES key) | Stored encrypted as `encrypted_data_key`; in memory / keychain after login |
| **Password hash** | Login verification only | SHA-256 of password + salt | Stored as `password_hash`; salt as `salt` |

---

## 3. Key Derivation (KEK from password)

- **Algorithm:** PBKDF2 (Password-Based Key Derivation Function 2).
- **Hash:** HMAC-SHA256 (`kCCPRFHmacAlgSHA256`).
- **Iterations:** 100,000.
- **Salt:** 32 bytes of cryptographically secure random data per user, stored as `key_salt` (e.g. base64).
- **Output:** 32-byte (256-bit) key used as the KEK (`SymmetricKey`).

So: **KEK = PBKDF2(password, key_salt, 100_000, HMAC-SHA256, 32 bytes)**.

---

## 4. Data Encryption Key (DEK)

- **Generation:** 256-bit random key via CryptoKit `SymmetricKey(size: .bits256)`.
- **Storage:** The DEK is encrypted with the KEK (see below). Only the **encrypted** DEK is stored in the database (`encrypted_data_key`). The key salt used for KEK derivation is stored in `key_salt`; it is **not** the AES-GCM nonce for DEK encryption.
- **Runtime:** After successful login, the DEK is derived (decrypted) once and can be cached in memory and/or stored in the keychain for the session.

---

## 5. Encrypting the DEK (KEK → encrypted DEK)

- **Cipher:** AES-GCM (CryptoKit).
- **Input:** Raw DEK bytes encrypted under the KEK.
- **Output:** The **combined** GCM format (nonce + ciphertext + authentication tag) is stored as `encrypted_data_key`. Decryption uses this combined blob; no separate nonce field is required for DEK decrypt.

---

## 6. Encrypting Measurement Data (DEK)

- **Plaintext:** `MeasurementData` encoded as JSON, then encrypted.
- **Cipher:** AES-GCM (CryptoKit).
- **Nonce:** Random nonce per encryption (generated by `AES.GCM.Nonce()`).
- **Storage:**  
  - Ciphertext (or combined GCM output) is stored as `encrypted_data` (BLOB).  
  - The nonce is stored separately as `encryption_iv` (e.g. base64 string) and is required for decryption.

Flow: **JSON(measurement) → encrypt with DEK and random nonce → store (ciphertext, nonce as iv)**.

---

## 7. Password Hashing (for login verification)

- **Purpose:** Verify the user’s password at login; not used directly as an encryption key.
- **Method:** SHA-256 over the UTF-8 bytes of `password + salt` (salt is a string, e.g. base64).
- **Storage:** The hex-encoded hash is stored as `password_hash`; the salt is stored as `salt`.

So: **password_hash = hex(SHA256(password || salt))** (with salt stored separately).

---

## 8. Randomness and salts

- **Salts:** 32-byte random values from `SecRandomCopyBytes(kSecRandomDefault, …)`.
- **Nonces:** AES-GCM nonces for measurement encryption are generated by CryptoKit (`AES.GCM.Nonce()`).
- **DEK:** 256-bit random key from CryptoKit.

---

## 9. Summary Table

| Component | Algorithm / method | Parameters / storage |
|-----------|--------------------|----------------------|
| KEK derivation | PBKDF2-HMAC-SHA256 | 100,000 iterations, 32-byte salt (`key_salt`), 32-byte key |
| DEK generation | Random | 256-bit CryptoKit `SymmetricKey` |
| DEK storage | AES-GCM (encrypt DEK with KEK) | Combined (nonce+ciphertext+tag) in `encrypted_data_key` |
| Measurement encryption | AES-GCM | DEK; nonce per encryption stored as `encryption_iv` |
| Password verification | SHA-256 | `password + salt` → hex; stored as `password_hash` and `salt` |

---

## 10. Data flow (conceptual)

1. **Registration / account creation**  
   Generate password salt and key salt → hash password (for storage) → derive KEK from password + key salt → generate DEK → encrypt DEK with KEK → store `password_hash`, `salt`, `key_salt`, `encrypted_data_key`. Optionally store DEK in keychain for current session.

2. **Login**  
   Verify password using stored `password_hash` and `salt` → derive KEK from password + `key_salt` → decrypt `encrypted_data_key` with KEK to get DEK → use DEK for measurement encrypt/decrypt; optionally cache DEK in keychain.

3. **Saving a measurement**  
   Encode measurement as JSON → encrypt with DEK (AES-GCM, new nonce) → store ciphertext and nonce (`encryption_iv`).

4. **Reading a measurement**  
   Load ciphertext and `encryption_iv` → decrypt with DEK (AES-GCM) → decode JSON to `MeasurementData`.

This document reflects the encryption methodology as implemented in the repository (e.g. `EncryptionService.swift`, `AuthenticationService`, and database usage of `encrypted_data_key`, `key_salt`, `encrypted_data`, `encryption_iv`).
